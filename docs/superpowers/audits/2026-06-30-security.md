# Security Audit — Galli (pages-mvp)
**Date:** 2026-06-30  
**Auditor:** Claude Code (automated, read-only)  
**Scope:** `src/`, `prisma/`, config files, git log

---

## Summary

The codebase demonstrates solid security fundamentals: JWT secret is validated at startup, bcrypt cost factor is 12, cookies are httpOnly + secure-in-production, authentication is consistently applied to sensitive mutations, Prisma ORM prevents SQL injection, and the custom HTML sanitizer covers all `dangerouslySetInnerHTML` usages. No hardcoded secrets were found in source or git history.

However, five meaningful issues require attention before go-live: two dependency upgrades (Next.js DoS, Vitest file-read), an IDOR in comment moderation, an unrevoked JWT on password reset, and SVG uploads reaching Vercel Blob without a Content-Security-Policy wrapper.

---

## Findings

---

### 1. Outdated Next.js — DoS via RSC Deserialization
**Severity:** High  
**Evidence:** `package.json` line — `"next": "^14.2.0"`. Audit output: *"Next.js HTTP request deserialization can lead to DoS when using insecure React Server Components — vulnerable >=13.0.0 <15.0.8"*  
**Why it matters:** A crafted HTTP request to any RSC endpoint can crash the server process, making the site unavailable.  
**Recommendation:** Upgrade Next.js to `>=15.0.8`. Test thoroughly — App Router API is stable across 14→15 but some config keys moved.

---

### 2. IDOR in Comment Moderation
**Severity:** High  
**Evidence:** `src/app/api/displays/[id]/comments/route.ts` — PATCH handler (line ~93):
```ts
const comment = await db.comment.update({
  where: { id: commentId },  // ← no check that comment.displayId === id
  data: { approved },
})
```
The handler verifies the caller owns display `[id]`, but then updates any `commentId` passed in the body — including comments belonging to other displays.  
**Why it matters:** Any authenticated user who owns at least one display can approve or reject comments on any other user's display by guessing or enumerating comment UUIDs.  
**Recommendation:** Add a `where` clause: `where: { id: commentId, displayId: id }`. If the comment doesn't belong to the display, Prisma will throw `RecordNotFound`.

---

### 3. SVG Upload Allowed — XSS via Direct Blob URL (Production)
**Severity:** High  
**Evidence:** `src/app/api/upload/route.ts` line 12: `'image/svg+xml'` is in `ALLOWED_TYPES`. In production (Vercel Blob path, line ~49), SVGs are stored and served directly from the Blob CDN with `contentType: file.type`, with no CSP header added by the application.  
**Why it matters:** SVG files can contain `<script>` tags. When a user navigates directly to the Blob CDN URL of a crafted SVG, the browser executes its JavaScript. While this is on the CDN domain (not galli.page), it can be used for phishing, credential harvesting via look-alike UI, or if Blob URLs share an origin, cookie theft.  
The local dev file-serve route (`src/app/api/upload/[...path]/route.ts`) adds a strict CSP that blocks script execution, but this protection does not extend to Vercel Blob responses.  
**Recommendation:** Remove `image/svg+xml` from `ALLOWED_TYPES`, or sanitize SVG content server-side (strip `<script>`, event handlers, `javascript:` hrefs) before storing. The safest option is to disallow SVG entirely.

---

### 4. Password Reset Does Not Invalidate Existing Sessions (JWT Revocation)
**Severity:** High  
**Evidence:** `src/app/api/auth/reset/route.ts` — updates `user.password` via Prisma. `src/lib/auth.ts` — JWT verification only checks signature + expiry; no generation/version field is stored on the user row.  
**Why it matters:** After a successful password reset (e.g., following account compromise), the attacker's existing `galli-auth` JWT remains valid for up to 7 days. The legitimate user's account is still accessible by the attacker.  
**Recommendation:** Add a `tokenVersion` integer column to `User`. Embed it in the JWT payload at sign-in. Increment it on password reset (and optionally on logout). In `getUser`/`verifyAuth`, reject tokens where `decoded.tokenVersion !== user.tokenVersion`.

---

### 5. Vitest Critical — Arbitrary File Read When UI Server Runs
**Severity:** High (dev-only risk, but critical CVSS)  
**Evidence:** `package.json` — `"vitest": "^4.0.18"`. Audit: *"When Vitest UI server is listening, arbitrary file can be read and executed — vulnerable >=4.0.0 <4.1.0"*  
**Why it matters:** If `vitest --ui` is run on a shared dev machine or CI with an exposed port, any file on the machine can be read/exfiltrated. Does not affect production (Vitest is a devDependency), but is a supply-chain risk for developers.  
**Recommendation:** Upgrade Vitest to `>=4.1.0`.

---

### 6. No Rate Limit on File Upload Endpoint
**Severity:** Medium  
**Evidence:** `src/app/api/upload/route.ts` — authenticated route, no `rateLimit()` call. Other sensitive routes (generate, login, signup) all apply rate limiting.  
**Why it matters:** An authenticated attacker can hammer the endpoint to exhaust Vercel Blob storage quota or bandwidth budget. At 10 MB/file this can be expensive.  
**Recommendation:** Add `rateLimit(request, { limit: 20, windowMs: 60_000, prefix: 'upload' })` before processing the form data.

---

### 7. Jersey Signature IP Hash Uses No Application Salt
**Severity:** Medium  
**Evidence:** `src/app/api/displays/[id]/signatures/route.ts` line ~51:
```ts
const ipHash = crypto.createHash('sha256').update(ip).digest('hex').slice(0, 16)
```
Compare with `src/app/api/forms/submit/route.ts` which correctly uses `ip + getJwtSecret()` as the hash input.  
**Why it matters:** IPv4 space is only ~4 billion addresses. A rainbow table of `SHA256(ip).slice(0,16)` for common IP ranges is trivially pre-computable, allowing reversal of stored IP hashes to identify signatories.  
**Recommendation:** Apply the same pattern as form submission: `crypto.createHash('sha256').update(ip + getJwtSecret()).digest('hex').slice(0, 16)`.

---

### 8. Analytics View Count Double-Incremented
**Severity:** Low  
**Evidence:** `src/app/api/displays/[id]/route.ts` increments `views` on every unauthenticated GET. `src/app/api/analytics/track/route.ts` also increments `views` when `eventType === 'view'`. Both are called from the public page.  
**Why it matters:** Display view counts are inflated 2× per real page visit, misleading owners about reach.  
**Recommendation:** Decide on a single source of truth for view counting. Remove the increment from the GET display route if the analytics track endpoint is the canonical source.

---

### 9. Comment Content Stored Unsanitized
**Severity:** Low  
**Evidence:** `src/app/api/displays/[id]/comments/route.ts` POST handler stores `content.trim()` directly. No sanitization is applied. Comments are later displayed in the analytics `ElementsTab` — how they are rendered needs verification.  
**Why it matters:** If any consumer renders comment content via `dangerouslySetInnerHTML` without sanitization, stored XSS is possible.  
**Recommendation:** Verify `src/components/analytics/element-cards/` renders comment content as plain text (not HTML). If any HTML rendering is needed, pipe through `sanitizeHtml()`.

---

### 10. No CORS Policy Configured
**Severity:** Low  
**Evidence:** No `next.config.*` headers block or API middleware setting `Access-Control-Allow-Origin`.  
**Why it matters:** By default Next.js API routes respond to cross-origin requests (though SameSite=Lax cookies block CSRF for cookie-authenticated endpoints). A missing CORS policy means any origin can read non-credentialed API responses (e.g., public display data, signatures, comments).  
**Recommendation:** Add an explicit `Access-Control-Allow-Origin` policy in `next.config.ts` for API routes. For non-public endpoints, block or restrict origins. For public endpoints (comments, signatures, analytics track) this is low risk.

## Launch Blockers (Critical / High)

1. **[High] Outdated Next.js (≥14.2, <15.0.8)** — DoS via RSC deserialization. Upgrade to 15.x before launch.
2. **[High] IDOR in comment moderation** — any display owner can moderate anyone's comments. One-line Prisma fix.
3. **[High] SVG upload → stored XSS via Vercel Blob CDN** — remove SVG from allowed types or sanitize before storage.
4. **[High] Password reset does not invalidate JWTs** — compromised sessions remain active 7 days post-reset. Add token version field.
5. **[High] Vitest critical CVE** — upgrade to >=4.1.0 (dev risk, trivial fix).
